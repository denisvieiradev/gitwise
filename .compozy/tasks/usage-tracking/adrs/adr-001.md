# ADR-001: Opt-in consent for usage telemetry

## Status

Accepted

## Date

2026-07-04

## Context

gitwise currently advertises a no-telemetry guarantee in `README.md` (Privacy section) and `SECURITY.md`: the only data leaving a user's machine is the diff sent to Claude. Adding usage tracking reverses that public commitment, so the consent posture is the highest-stakes decision in this feature.

Two industry patterns exist for CLI telemetry:

- **Opt-out** (on by default), used by the GitHub CLI and the .NET SDK.
- **Opt-in** (off by default), used by Cargo (which collects nothing) and favored by the developer-tool community.

When the GitHub CLI enabled opt-out telemetry by default in 2026, it drew significant public backlash. gitwise's audience is developers — the demographic most sensitive to default-on tracking — and the tool markets "no telemetry" as a feature.

## Decision

Usage telemetry is **opt-in and off by default**. Nothing is collected until the user explicitly enables it. A friendly one-time prompt on the CLI first-run flow asks the user to enable it, explaining what is and is not collected. Consent is stored as a shared flag in `~/.gitwise/config.json` and honored by every surface (CLI, Claude Code plugin, Gemini mirrors). The published privacy language changes from "no telemetry" to "optional telemetry, off by default."

## Alternatives Considered

### Alternative 1: Opt-out (on by default)

- **Description**: Collect from first run with a one-time notice and an easy off-switch.
- **Pros**: Near-complete data; census-level signal for roadmap decisions.
- **Cons**: Reverses a public written promise; high reputational risk with a developer audience.
- **Why rejected**: The trust cost of reversing a marketed "no telemetry" promise is not easily reversible, whereas opt-in → opt-out remains an option later if enrollment proves too low.

### Alternative 2: Silent opt-in (config flag only, no prompt)

- **Description**: Off by default, enabled only if the user sets a flag or env var, with no prompt.
- **Pros**: Maximum privacy; no prompt noise.
- **Cons**: Near-zero enrollment; effectively no data.
- **Why rejected**: A prominent first-run prompt lifts enrollment meaningfully above a buried flag while keeping consent genuine.

## Consequences

### Positive

- Preserves the no-telemetry brand as "optional, off by default."
- Zero backlash risk; aligns with developer-tool community norms.
- A later move to opt-out remains possible if enrollment is too low.

### Negative

- Realistic enrollment of ~5–10% yields a small, self-selected sample; weak for precise quantitative analysis.

### Risks

- **Low enrollment starves the data.** Mitigation: make the first-run prompt clear and value-oriented ("help improve gitwise"); revisit posture after measuring real enrollment.

## Implementation Notes

Consent flag lives in the existing `~/.gitwise/config.json`. Env override `GITWISE_TELEMETRY=0/1` supports non-interactive/CI contexts. The plugin and Gemini surfaces read the same flag and stay silent until it is enabled.
