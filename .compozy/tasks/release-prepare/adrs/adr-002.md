# ADR-002: Minimal release-scoped strategy abstraction, not a full FlowStrategy

## Status

Accepted

## Date

2026-05-18

## Context

The Devflow CLI Pivot design doc (`2026-04-02-gitflow-pivot-design.md`) proposes a project-wide `FlowStrategy` interface covering branch naming, PR targets, merge validation, **and** release flow, with separate implementations for GitFlow, GitHub Flow, GitLab Flow, and Trunk-based. That abstraction touches every command (`commit`, `pr`, `release`, `merge`, `worktree`, `init`).

The current task is scoped to `release-prepare` only. Pulling the full FlowStrategy in now would:

- Force `commit`, `pr`, and `review` to take a dependency on a strategy that doesn't influence their behavior yet.
- Block this task on the broader pivot effort (new `init` flow, new config schema, new `merge` and `worktree` commands).
- Introduce extension points (`getBranchName`, `validateBranch`, `canMergeTo`, `getMergeMethod`) that have no caller in this task â€” violating YAGNI.

But we still need *some* strategy awareness in release, because GitFlow's release flow is fundamentally different from GitHub Flow's: branch creation, multiple merge targets, conditional develop existence.

## Decision

Introduce a narrow `ReleaseStrategy` interface scoped exclusively to the release lifecycle, with two implementations: `github-flow` (default) and `gitflow`. Other strategies from the design doc are explicitly out of scope for this task.

```typescript
// packages/core/src/strategies/release.ts

export type ReleaseStrategyName = "github-flow" | "gitflow";

export interface PrepareContext {
  cwd: string;
  newVersion: string;
  develop?: string;  // resolved from config; undefined for non-GitFlow
  main: string;
}

export interface FinishContext extends PrepareContext {
  tag: string;
}

export interface ReleaseStrategy {
  readonly name: ReleaseStrategyName;
  /** Optional release branch to create during prepare; null = no branch */
  releaseBranchFor(version: string): string | null;
  /** Branches to merge the release into during finish, in order */
  mergeTargets(): string[];
  /** True if `develop` must exist for this strategy to run */
  requiresDevelop(): boolean;
}
```

The strategy is selected by reading `RepoConfig.releaseStrategy` (new optional field). Unset = `github-flow`, preserving today's behavior. A factory `createReleaseStrategy(name)` returns the right implementation; everything else lives in `release.ts`.

We deliberately do **not** introduce `BranchType`, `MergeMethod`, `PRTarget`, `validateBranch`, or strategy-driven branch naming â€” those belong to the future `commit`/`pr`/`merge` tasks.

## Alternatives Considered

### Alternative 1: Adopt the full FlowStrategy from the design doc now

- **Description**: Implement the entire `FlowStrategy` interface in this task â€” branch naming, PR targets, merge rules, the four strategies.
- **Pros**: One large change instead of two; matches the long-term design exactly.
- **Cons**: 5x the surface area for code review; most of the interface is unused until later tasks land; risks landing a design that later tasks reveal is wrong; blocks `release-prepare` on unrelated work.
- **Why rejected**: YAGNI. We can extend the interface (or replace it with `FlowStrategy`) when the broader pivot actually arrives; doing it now is speculative.

### Alternative 2: No abstraction, branch on `config.releaseStrategy === "gitflow"` inside `release.ts`

- **Description**: Inline `if (cfg === "gitflow") { ... } else { ... }` blocks directly in the release functions.
- **Pros**: Less code; trivially obvious for two strategies.
- **Cons**: Two strategies today, four likely tomorrow (GitLab Flow, trunk-based at minimum); inline branching duplicates the same shape in `prepare` and `finish`; harder to unit-test the strategy rules in isolation.
- **Why rejected**: The two-strategy footprint is enough to justify a tiny interface, and isolating the rules in `strategies/release.ts` makes each one a single object with three small methods â€” trivially testable.

### Alternative 3: Make strategy a runtime flag (`--strategy gitflow`) instead of config

- **Description**: Drop the config field, pass strategy on every invocation.
- **Pros**: No config-schema change.
- **Cons**: Users must remember the flag every time; `gw release prepare` and `gw release finish` must agree on it â€” easy to forget on `finish` and end up tagging without merging to develop.
- **Why rejected**: Strategy is a repo-level property, not a per-invocation choice. Config is the right place.

## Consequences

### Positive

- Adds ~30 lines of strategy code total (interface + two implementations + factory).
- Repos without a `releaseStrategy` config keep behaving exactly as today.
- The interface is small enough to expand or replace later without a deprecation cycle.

### Negative

- A future broader strategy abstraction will likely subsume this one, meaning a small follow-up rename when the design doc's `FlowStrategy` lands.
- Two strategy names live in the config schema before all four exist.

### Risks

- **Drift from the design doc when the broader pivot lands**: mitigated by keeping `ReleaseStrategyName` a closed string union and re-exporting from a single file; rename/replace is a one-day refactor.
- **GitFlow `develop` branch missing**: `requiresDevelop()` is checked in `prepare`; `release-prepare` fails fast with a clear error pointing to `git checkout -b develop`.

## Implementation Notes

- New file: `packages/core/src/strategies/release.ts` â€” interface, factory, two implementations, ~80 lines total.
- Config additions: `RepoConfig.releaseStrategy?: ReleaseStrategyName` and `RepoConfig.developBranch?: string` (defaults to `"develop"`).
- The `name` field on `ReleaseStrategy` is for logging / telemetry only; callers use the methods.
- Strategies are stateless singletons â€” `createReleaseStrategy` returns a pre-built instance, not a new one.
