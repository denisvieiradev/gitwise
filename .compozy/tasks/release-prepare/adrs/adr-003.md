# ADR-003: Release plan file lifecycle and integrity checks

## Status

Accepted

## Date

2026-05-18

## Context

ADR-001 introduces `.gitwise/release-plan.json` as the handoff between `prepare` and `finish`. A plan file that sticks around after `finish`, or one that doesn't match the current repo state, is worse than no plan at all — it leads to double-tags, version mismatches, or a `finish` that creates a tag for a release that never actually happened.

We need explicit rules for: when the file is written, what it contains, when it is deleted, and what `finish` checks before trusting it.

## Decision

The plan file is **created last** in `prepare` and **deleted first** in `finish`, and `finish` validates the plan against live repo state before performing any mutation.

### File contents

```typescript
// packages/core/src/commands/release-plan.ts
export interface PersistedReleasePlan {
  /** Schema version for forward-compatible reads */
  schema: 1;
  /** Strategy that produced this plan */
  strategy: ReleaseStrategyName;
  /** Versions */
  currentVersion: string;
  newVersion: string;
  suggestedBump: BumpType;
  /** Generated artifacts */
  changelog: string;
  notes: string;
  commits: string;
  /** Repo state at prepare time */
  preparedAt: string;          // ISO-8601
  baseCommit: string;          // SHA at HEAD when prepare ran
  targetBranch: string;        // branch finish must complete on
  releaseBranchCreated: boolean;
  /** Token usage for transparency */
  tokens: { input: number; output: number };
}
```

### Lifecycle invariants

1. **`prepare` writes the plan last.** Order: ensure `.gitwise/` dir → write `release-X.Y.Z.md` notes → mutate `package.json` and `CHANGELOG.md` only if `releaseBranchCreated` (GitFlow) → write `release-plan.json`. If any earlier step throws, the plan file is never created.
2. **`finish` deletes the plan first.** Order: load plan → validate → delete plan file → perform commits/tags/push/gh-release. If validation fails, the file is left in place (the user inspects or `abort`s). If a mutation step fails after deletion, the user can recover with git ops; the plan won't trigger a second `finish`.
3. **`abort` deletes the plan and optionally the release branch.** Asks before deleting the branch; refuses if the branch has commits not present in `main` or `develop`.
4. **`.gitwise/release-plan.json` is added to `.gitignore`** on first `prepare` if not already covered. We never want this file in a commit.

### Validation in `finish`

`finish` MUST reject the plan with a specific error when any of:

- File schema is not `1`.
- `plan.newVersion` is already tagged (`v<newVersion>` exists).
- For GitFlow: current branch is not `plan.targetBranch` (the release branch).
- For GitFlow: configured `developBranch` no longer exists.
- Working tree is dirty (existing `WORKING_TREE_DIRTY` check is reused).

Errors are typed (`STALE_PLAN_TAG_EXISTS`, `STALE_PLAN_BRANCH_MISMATCH`, etc.) so the CLI can render targeted recovery hints.

## Alternatives Considered

### Alternative 1: Write plan first, then mutate files

- **Description**: Save `release-plan.json` immediately after generating it, then bump manifests and write notes.
- **Pros**: Plan is durable even if the rest of prepare crashes — user can resume.
- **Cons**: A crash mid-prepare leaves the plan claiming `releaseBranchCreated: true` when only half the work happened; `finish` would proceed against an inconsistent state.
- **Why rejected**: Atomicity matters more than resume-ability for a step that takes seconds. If prepare crashes, re-running is cheap and clean.

### Alternative 2: Validate by re-running `release()` and diffing

- **Description**: On `finish`, re-plan and check the new plan matches the persisted one.
- **Pros**: Catches subtle drift (new commits since prepare).
- **Cons**: Doubles the LLM cost; non-deterministic LLM output triggers false positives; defeats the point of ADR-001.
- **Why rejected**: We already store `baseCommit`; the cheap check (`git log baseCommit..HEAD`) gives the user the information without re-planning. We surface "new commits since prepare" as a warning, not a hard failure.

### Alternative 3: No schema version

- **Description**: Trust JSON.parse, assume current shape forever.
- **Pros**: Less code.
- **Cons**: A future change to the plan shape silently corrupts in-flight releases for users who upgrade gitwise between `prepare` and `finish`.
- **Why rejected**: `schema: 1` is a one-line check that buys forward compatibility.

## Consequences

### Positive

- Predictable plan-file lifecycle: only ever exists between a successful `prepare` and the first `finish`/`abort`.
- Drift between prepare and finish surfaces as typed errors with actionable recovery, not silent wrong-version tags.
- Forward-compatible: schema version lets v2 plans coexist with v1.

### Negative

- More error codes for the CLI to handle (worth it — the existing release flow already exports `WORKING_TREE_DIRTY` and `TAG_EXISTS` in the same style).

### Risks

- **User edits `release-plan.json` by hand**: not protected; documented as "do not edit, edit the release-notes file instead."
- **`.gitignore` modification surprises the user**: only happens on first `prepare`; we print a one-line notice when we add the entry.

## Implementation Notes

- New file: `packages/core/src/commands/release-plan.ts` — `PersistedReleasePlan` type, `saveReleasePlan`, `loadReleasePlan`, `deleteReleasePlan`, `ensureGitignored`.
- All four functions are pure async filesystem helpers; no LLM or git deps.
- `ensureGitignored` checks for `.gitwise/release-plan.json` or a covering wildcard (`.gitwise/`, `.gitwise/*`) before appending.
- Existing `applyRelease()` is refactored to consume a `PersistedReleasePlan` rather than the in-memory `ReleasePlan`; the legacy one-shot path constructs the persisted form in memory and passes it through, so the code path is unified.
