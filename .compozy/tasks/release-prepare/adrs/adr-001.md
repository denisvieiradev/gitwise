# ADR-001: Split `release` into explicit `prepare` and `finish` subcommands

## Status

Accepted

## Date

2026-05-18

## Context

The current `release()` core function plans a release (LLM-generated bump, changelog, notes) and immediately returns the plan. The CLI then asks for confirmation and calls `applyRelease()` in the same process. This works for GitHub-flow / trunk-based workflows where a release is a tag on `main`, but it does not fit GitFlow:

- GitFlow requires creating a long-lived `release/X.Y.Z` branch, bumping version + writing changelog on that branch, **stabilizing** there (additional commits, manual QA, hotfixes), and only then merging into `main` and `develop` with a tag.
- The stabilization phase happens across **multiple sessions** and possibly multiple people. A single-process plan-then-apply cannot model it.
- Even for non-GitFlow projects, users routinely want to inspect the generated changelog and notes, edit them, and run `release` again â€” today they re-run the whole flow, paying the LLM cost twice.

We also need to preserve the current one-shot UX (`gw release` â†’ confirm â†’ done) because it covers the majority of solo-OSS / indie-developer use cases described in the gitwise PRD.

## Decision

Introduce two explicit subcommands and persist the plan between them:

- `gw release prepare [version]` â€” runs the existing planning logic, optionally creates a release branch (per strategy â€” see ADR-002), writes the new version + changelog to disk, **and stops**. Persists the plan as `.gitwise/release-plan.json`. Does **not** commit, tag, push, or create a GitHub release.
- `gw release finish` â€” loads `.gitwise/release-plan.json`, performs strategy-specific merge moves (GitFlow: merge into `main` and `develop`), tags, pushes, creates the GitHub release, and deletes the plan file.
- `gw release abort` â€” deletes `.gitwise/release-plan.json` and, when a release branch was created, prompts before deleting it.
- `gw release` (no subcommand) keeps today's behavior: `prepare` + interactive confirm + `finish` in a single process, with the plan file written and immediately consumed. This is the default for backwards compatibility.

The plan file is the **single source of truth** between phases. It is added to `.gitignore` so it never leaks into commits.

## Alternatives Considered

### Alternative 1: Two flags on a single `release` command (`--prepare-only`, `--finish`)

- **Description**: Keep one `release` command, use flags to control phase.
- **Pros**: Smaller surface area; less wiring in the Commander program.
- **Cons**: Subcommands surface the workflow in `gw release --help`; flag combinations create error states (`--prepare-only` + `--finish` is meaningless); doesn't extend cleanly to `abort`.
- **Why rejected**: Subcommands document the intended workflow better and match the language users already use ("release prepare", "release finish") in GitFlow tooling.

### Alternative 2: Stateless re-plan on finish (no plan file)

- **Description**: `finish` re-runs the LLM planner against the same commit range to recreate the plan.
- **Pros**: No disk state; no cleanup; no possibility of stale plans.
- **Cons**: Double-pays the LLM cost; the regenerated changelog/notes can drift from what the user already approved; user-edited release notes on disk are lost; cannot model a release branch that has accumulated extra commits since prepare.
- **Why rejected**: The whole point of `prepare` is to lock in an approved plan so the user can stabilize and then ship the exact thing they reviewed.

### Alternative 3: External state in a hidden git ref (`refs/gitwise/release-plan`)

- **Description**: Store the plan as a serialized blob behind a custom git ref.
- **Pros**: Travels with the repo; survives clones.
- **Cons**: Heavier infrastructure; harder to inspect and edit; collides with other tools that read all refs; adds a new git surface area we'd have to teach users about.
- **Why rejected**: Overkill for a file that should be local-only and short-lived. `.gitwise/release-plan.json` (gitignored) is simpler and inspectable with `cat`.

## Consequences

### Positive

- GitFlow projects get a workflow that matches their actual release model.
- LLM costs are paid once per release, not on every retry of `apply`.
- Users can manually edit `.gitwise/release-notes-X.Y.Z.md` between `prepare` and `finish`; `finish` consumes the on-disk file rather than the in-memory plan field.
- The default `gw release` invocation behaves exactly as today, so no migration is required for existing users.

### Negative

- Two extra commands to document and test.
- A new on-disk state file introduces a new error mode (stale plan, plan-for-wrong-branch).

### Risks

- **Stale plan after a long stabilization window**: mitigated by storing the `currentVersion`, `targetBranch`, and `baseCommit` SHA in the plan; `finish` refuses to run if the working state has drifted (e.g. version already bumped, tag exists, branch deleted).
- **Plan written but `prepare` crashes before completing branch creation**: mitigated by writing the plan file **last**, after branch + manifest writes succeed. A crash mid-prepare leaves either no plan file (user re-runs cleanly) or a plan plus a partially-set-up branch (`abort` cleans both).
- **Two operators try to prepare on the same repo**: not protected â€” same constraint as any local-only tool. Documented in the command help.

## Implementation Notes

- Plan file path: `<cwd>/.gitwise/release-plan.json` (alongside the existing `.gitwise/release-X.Y.Z.md` notes).
- The plan file MUST be added to `.gitignore` on first prepare if not already present.
- `finish` validates: file exists, `plan.newVersion` is not already tagged, current branch matches `plan.targetBranch` for non-strategy invocations.
- Add a `core` export `loadReleasePlan(cwd) â†’ ReleasePlan | null` and `saveReleasePlan(cwd, plan)` so both CLI and skills surfaces can drive the workflow.
