## Shared Decisions

- `ReleaseStrategy.mergeTargets(mainBranch, developBranch?)` follows the TechSpec signature, not the parameter-less ADR-002 sketch. Callers should pass the actual repo main branch name (not the literal `"main"`). github-flow ignores develop; gitflow returns `[mainBranch, developBranch]` in that order, or `[mainBranch]` if develop is omitted (gitflow consumers must always supply it — `requiresDevelop()` is `true`).
- Strategies are stateless and shared: `createReleaseStrategy("github-flow")` returns the same instance across every call (reference equality). Downstream code is free to memoize the result or call the factory each time — both are cheap.
- `deepMerge` uses conditional spreads — new repo-config fields that must surface in `MergedConfig` need the same `...(override.x !== undefined && { x: override.x })` pattern AND a parallel optional declaration on `MergedConfig`.

## Shared Learnings

- `readRepoConfig` does not validate field names — it accepts whatever JSON parses; `INVALID_REPO_CONFIG` fires only on malformed JSON. New optional fields therefore need no allowlist work.

- Task 06 (`finishRelease`) consumes the plan via `loadReleasePlan`. For gitflow, the release branch already carries a committed `chore(release): v<version>` containing the package.json bump + CHANGELOG entry + the `.gitignore` line (prepare commits these in a single commit on the release branch). For github-flow, prepare did NOT mutate `package.json` / CHANGELOG — `finish` does those mutations + commit before tagging. `finishRelease.workspacePropagation` (new in task_08) runs after the root bump and before the commit on the github-flow branch; the same packages-add step from the legacy `applyRelease` is preserved.
- Task 06 also relies on `plan.targetBranch` — for gitflow it's `release/<v>`, for github-flow it's whatever branch the user was on when prepare ran (typically `main`). ADR-003 lifecycle is implemented exactly: `loadReleasePlan` → validate → reload notes from disk → `deleteReleasePlan` FIRST → (gh-flow only) bump+commit → strategy-driven merges (self-merge skipped) → checkout main → tag + pushWithTags + push develop → optional `gh release` (graceful) → safe `git branch -d` of the release branch.
- Task 05 / 06 / 07 / 08 consume the plan via `import { saveReleasePlan, loadReleasePlan, deleteReleasePlan, ensureGitignored, type PersistedReleasePlan } from "./release-plan.js"`. `loadReleasePlan` returns `null` for absent plan (callers must throw `NO_RELEASE_PLAN`), throws `INVALID_PLAN_SCHEMA` for schema mismatch / non-object JSON, and `INVALID_PLAN_JSON` for malformed JSON — handle these three cases distinctly in the CLI error renderer (task_09).
- `finishRelease`'s dirty-tree check FILTERS the prepare-emitted paths (`.gitwise/`, `.gitwise/release-plan.json`, `.gitwise/release-<v>.md`, `.gitignore`). Task_08's legacy one-shot path drives both phases in-process, so the same filter (or the same `finishRelease` call) must be used — otherwise the dirty check trips on legitimate prepare state.
- `prepareRelease` reads `RepoConfig` internally via `readRepoConfig(cwd)`. `finishRelease` does the same for `developBranch`, but it has NO opts override (TechSpec FinishReleaseOptions = `{ cwd, tagAndPush?, createGhRelease?, deleteReleaseBranch? }`). Users who configure a non-default develop branch must do so in `.gitwise.json`. Default chain on prepare: opts → RepoConfig → `"github-flow"` (strategy) / `"develop"` (developBranch).
- `git.branchExists` swallows the non-zero exit and returns `false` for missing refs — callers should NOT wrap it in try/catch.
- `prepareRelease` captures `baseCommit` AFTER preflight but BEFORE the LLM call and any branch creation. For gitflow this is the user's branch HEAD when they invoked prepare (could be develop, could be a feature branch — gitflow does NOT require the user to be on develop). Useful for finish to warn on drift, but not as a merge base.
- `abortRelease` returns `RELEASE_BRANCH_UNMERGED` (NOT in the TechSpec's enumerated error list). The CLI in task_09 must render this code with a recovery hint pointing the user at either merging into the surfaced target or removing the branch by hand. The error message already includes the unmerged target branch name in quotes.
- Test runs in this repo MUST go through `npm test` (which sets `--experimental-vm-modules`). `npx jest` directly fails on six suites due to a pre-existing `import.meta` ts-jest config issue on `loader.ts`. `npm run lint` at the package level is just `tsc --noEmit`; the root-level `eslint` is broken by a missing `jiti` dep in the workspace's node_modules.
- Commander v12 hoists same-named options to the parent. When a subcommand declares an option with the same long name as one declared on its parent (e.g. both `release` and `release prepare` declare `--bump`), the value lands in the parent's `opts()` and the subcommand's own `opts()` is empty. Subcommand action handlers must read `cmd.optsWithGlobals()` (which merges parent + subcommand) — or drop the option from one side. Affects `--bump`, `--no-gh-release`, `--no-workspace-propagation` between root `release` and the prepare/finish subcommands.
- CLI typecheck (`tsc --noEmit` in `packages/cli`) consumes the BUILT core dist at `packages/core/dist/index.d.ts`, not the source. After adding/renaming exports in `packages/core/src/index.ts`, run `npm run --workspace=@denisvieiradev/gitwise-core build` before `npm run typecheck` or CLI typecheck will fail with stale `TS2305: Module … has no exported member …` errors.
- `git tag -l --format=%(contents)` strips lines that start with `#` from annotated-tag messages (git treats them as comments by default). Tests asserting on tag annotation contents should either use non-`#` headings or pre-strip the comments themselves.
- `git.run()` in `packages/core/src/infra/git.ts` calls `stdout.trim()`, which is destructive for porcelain-status output: unstaged-modified lines like `" M filename"` lose their leading space and downstream parsers using `line.slice(3)` then misread the path (drops the first filename char). `git.status()` now strips only trailing newlines for that reason — any future helper that returns format-sensitive output should bypass `run()` the same way.
- The bundled-template path in `packages/core/src/template/loader.ts` must accommodate BOTH the source layout (`src/template/loader.ts` → `../../templates`) and the tsup dist layout (`dist/index.js` → `../templates`). The loader now probes both candidates. Any new bundled-asset path in the package needs the same dual-layout handling or it will work in jest (ts-jest reads source) and fail under `node dist/index.js`.
