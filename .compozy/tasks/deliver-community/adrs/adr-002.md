# ADR-002: Automated security and dependency gates in CI

## Status

Accepted

## Date

2026-05-21

## Context

The current `.github/workflows/ci.yml` runs lint, build, and test on Node 18/20/22. It does not run any static analysis for security issues, does not monitor dependencies for known vulnerabilities, and does not flag outdated dependency versions. As `gitwise` is positioned for community delivery and runs on user machines invoking git, subprocess injection or vulnerable dependencies are the most likely vectors a contributor (or a compromised transitive dep) could introduce a regression.

The codebase audit confirmed there are currently no shell-injection sites, but the lack of automated gates means a future PR could introduce one and pass CI.

## Decision

Add three security gates to the CI pipeline, each running in its own workflow file with clear ownership:

1. **CodeQL** — `.github/workflows/codeql.yml`. JavaScript/TypeScript pack, default queries plus `security-and-quality` and `security-extended`. Runs on every PR to `main` and on a weekly schedule. Failures block merge.
2. **Dependabot** — `.github/dependabot.yml`. Two ecosystems: `npm` (grouped by major/minor/patch, weekly) and `github-actions` (weekly, pinned-SHA updates). Auto-merge is enabled for patch and minor `npm` updates via a separate `.github/workflows/dependabot-auto-merge.yml` that requires the test matrix to pass.
3. **OSV-Scanner** — `.github/workflows/osv-scanner.yml`. Runs daily and on every PR. Fails the workflow on HIGH or CRITICAL vulnerabilities. Findings without an available fix can be temporarily acknowledged via a checked-in `osv-scanner.toml` ignore file with an expiry date enforced by the workflow.

OSSF Scorecard was explicitly excluded from this round at the user's direction; it can be added later as a public-facing reputation signal once the underlying controls have stabilized.

## Alternatives Considered

### Alternative 1: Add only CodeQL

- **Description**: Just SAST. Skip dependency monitoring.
- **Pros**: Lowest CI cost. Catches the most-impactful code-level issues.
- **Cons**: Misses the dominant supply-chain vector (vulnerable transitive deps). gitwise has 200+ transitive deps; without monitoring, a critical CVE could land silently.
- **Why rejected**: Vulnerable dependencies are the single most-common path to compromise for JS tooling.

### Alternative 2: Use Snyk or another third-party SaaS

- **Description**: Replace OSV-Scanner with a hosted alternative (Snyk, Socket, GitGuardian).
- **Pros**: Richer UI, additional checks (typosquatting, malicious packages).
- **Cons**: Requires another account, adds a billing dependency, sends repo metadata to a third party. OSV-Scanner is OSS, GitHub-runnable, and consumes the OSV.dev feed used by GitHub Advisory Database itself.
- **Why rejected**: For an OSS project, the OSS-native tool is the appropriate floor. Third-party SaaS is a future consideration if findings volume justifies it.

### Alternative 3: Use `npm audit` directly

- **Description**: Run `npm audit --audit-level=high` as a CI step.
- **Pros**: Zero new tooling.
- **Cons**: `npm audit` produces noisy results, the database is narrower than OSV, and there is no machine-friendly way to acknowledge unfixable findings.
- **Why rejected**: OSV-Scanner with the ignore file gives actionable signal; `npm audit` does not.

## Consequences

### Positive

- A future PR that introduces a child_process call with `shell: true` will be flagged by CodeQL.
- Dependency CVEs are detected within 24 hours of disclosure.
- Major dependency updates surface as Dependabot PRs that the maintainer reviews; minor and patch updates can flow with low overhead.
- The full graph of pinned vs. dynamic dependencies is visible from the Security tab.

### Negative

- CI runtime increases by roughly 3–5 minutes per PR (CodeQL is the dominant cost).
- Dependabot generates noise; the auto-merge workflow plus grouping mitigate this but it is not zero.
- OSV-Scanner false positives or unfixable findings require manual ignore-list maintenance.

### Risks

- **Risk**: CodeQL flags a finding in a release-critical hotfix path, blocking shipping. **Mitigation**: hotfix branches can request a single-PR exception documented in `CONTRIBUTING.md`; the issue must be filed and addressed within the next release cycle.
- **Risk**: Auto-merge on patch deps masks a malicious release of a legitimate package. **Mitigation**: full test matrix must pass before merge, and `osv-scanner` runs on every PR including Dependabot's — a malicious package with a published advisory would be caught.
- **Risk**: OSV ignore-file expiries are skipped if maintenance lapses. **Mitigation**: the workflow fails the build when an ignore entry expires, forcing review.

## Implementation Notes

- CodeQL setup uses the default `github/codeql-action/init` + `analyze` pair. Pin to a tagged SHA per ADR-001.
- Dependabot grouping config:
  ```yaml
  groups:
    npm-minor-and-patch:
      patterns: ["*"]
      update-types: ["minor", "patch"]
  ```
- The auto-merge workflow uses `gh pr merge --auto --squash` only when the actor is `dependabot[bot]` and update type is `version-update:semver-patch` or `version-update:semver-minor`.
- OSV-Scanner is invoked via `google/osv-scanner-action`.
