# ADR-003: GitwiseError class with stable exit codes and machine-readable error envelope

## Status

Accepted

## Date

2026-05-21

## Context

The codebase audit found that errors are currently thrown as `Object.assign(new Error(msg), { code: "ERROR_CODE" })`. The `code` field is used internally for control flow but is not surfaced to the CLI caller — `packages/cli/src/index.ts` prints the message and exits 1 for every failure. Consumers who want to script around `gw` cannot distinguish "nothing staged" from "API key missing" from "git push failed" without parsing English text.

For an open-source CLI that downstream tools will compose into shell pipelines, GitHub Actions, and pre-commit hooks, stable exit codes and machine-readable error output are a usability requirement, not a nice-to-have. They are also a precondition for the transactional rollback work in [[adr-004]] — rollback decisions depend on knowing which step failed and why.

## Decision

Introduce a `GitwiseError` class in `@denisvieiradev/gitwise-core` and standardize the CLI's error surface around it.

```ts
export class GitwiseError extends Error {
  readonly code: string;
  readonly exitCode: number;
  readonly cause?: unknown;
  readonly details?: Record<string, unknown>;

  constructor(args: { code: string; message: string; exitCode?: number; cause?: unknown; details?: Record<string, unknown> }) {
    super(args.message);
    this.name = "GitwiseError";
    this.code = args.code;
    this.exitCode = args.exitCode ?? EXIT_CODES[args.code] ?? 1;
    this.cause = args.cause;
    this.details = args.details;
  }
}
```

A `EXIT_CODES` table is exported from core and documented in `docs/exit-codes.md`. Initial allocation:

| Code | Constant | Category |
|------|----------|----------|
| 0 | `OK` | Success |
| 1 | `UNKNOWN` | Catch-all fallback |
| 10 | `NOTHING_STAGED` | No staged changes for `gw commit` |
| 11 | `INVALID_INTENT` | Unparseable `--intent` |
| 20 | `GIT_FAILED` | Underlying `git` command failed |
| 21 | `GH_FAILED` | Underlying `gh` command failed |
| 22 | `REPO_STATE_INVALID` | Detached HEAD, no upstream, conflicting branch |
| 30 | `API_FAILED` | Anthropic API or Claude binary call failed |
| 31 | `API_KEY_MISSING` | `ANTHROPIC_API_KEY` not configured |
| 32 | `API_RATE_LIMITED` | After max retries |
| 40 | `USER_ABORT` | User declined a confirmation prompt |
| 50 | `CONFIG_INVALID` | `.gitwise.json` malformed |
| 60 | `RELEASE_PLAN_STALE` | Plan file does not match current HEAD |
| 61 | `RELEASE_BRANCH_CONFLICT` | Pre-existing release branch blocks prepare |
| 70 | `SENSITIVE_FILE_BLOCKED` | Staged file matches sensitive-file pattern |

CLI changes (`packages/cli/src/index.ts`):
- Errors of type `GitwiseError` exit with `err.exitCode`; everything else exits 1 and is logged as `UNKNOWN`.
- A global `--json` flag emits errors as `{"error": {"code": "...", "message": "...", "exitCode": N}}` to stdout, exiting normally with the same exit code.
- Stderr always shows the human message and a one-line hint pointing at `docs/exit-codes.md`.

## Alternatives Considered

### Alternative 1: Keep current pattern, just document codes

- **Description**: Document the existing `code` strings; all CLI failures continue to exit 1.
- **Pros**: Zero refactor. Backwards-compatible.
- **Cons**: Shell scripting users cannot branch on failures. Defeats half of the community-readiness goal.
- **Why rejected**: Documentation without enforcement leaves the contract unverifiable; future code may stop setting `code` and nothing breaks.

### Alternative 2: Result<T, E> pattern across core

- **Description**: Refactor command pipeline to return `Result<T, GitwiseError>` instead of throwing.
- **Pros**: Most idiomatic for TS, eliminates accidental unhandled rejections, makes error paths visible at every call site.
- **Cons**: Touches every command file and every test. Months of refactor work. Existing tests assume thrown errors.
- **Why rejected**: The benefit over a disciplined throw-with-typed-error is small, the cost is large, and the refactor would block other 1.0 hardening work.

### Alternative 3: Use Node's `error.code` convention with no class

- **Description**: Throw `Error` instances with a `code` property and a documented exit-code mapping. No subclass.
- **Pros**: Aligns with Node core idiom (e.g., `ENOENT`).
- **Cons**: No `instanceof` check; downstream code can't distinguish a gitwise error from any other thrown error. `--json` mode and the `details` field need a structural typeguard which is fragile.
- **Why rejected**: A class costs almost nothing to introduce and unlocks reliable distinction.

## Consequences

### Positive

- Shell-script integrations gain a stable contract: `gw commit; if [ $? -eq 10 ]; then ...`.
- `--json` mode makes gitwise composable with task runners and CI systems.
- The rollback logic in [[adr-004]] can dispatch on `code` to decide compensating actions.
- Tests can assert on `code` instead of brittle message substring matches.

### Negative

- Existing internal call sites need migration from `Object.assign` to `new GitwiseError(...)`. Roughly 40 sites in core, 8 in cli.
- Exit-code table becomes a public contract; renumbering is a breaking change. Reserve unused numbers for future categories.

### Risks

- **Risk**: A code is introduced internally without being added to the documented table, drifting from the contract. **Mitigation**: a unit test reads `docs/exit-codes.md`, parses the table, and asserts every code in `EXIT_CODES` is documented and vice versa.
- **Risk**: A non-`GitwiseError` leaks (e.g., a raw git stderr buffer). **Mitigation**: the global handler in `cli/src/index.ts` always wraps unknown errors into `GitwiseError({ code: "UNKNOWN", ... })`.

## Implementation Notes

- Add `packages/core/src/errors.ts` exporting `GitwiseError`, `EXIT_CODES`, and a `wrapError(err: unknown): GitwiseError` helper.
- Add a codemod-style PR to convert existing `Object.assign(new Error(...), { code })` sites in one pass; do not mix with feature work.
- Add `docs/exit-codes.md` consuming the table programmatically (or check it in as a hand-maintained doc with a parity test).
- Add a `gw --version` JSON output that mirrors `--json` mode for consistency.
