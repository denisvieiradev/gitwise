# ADR-005: BDFL governance with CODEOWNERS and Contributor Covenant

## Status

Accepted

## Date

2026-05-21

## Context

`gitwise` is presently maintained by a single person (denisvieira05@gmail.com). `SECURITY.md` already names this person as the security contact, but the repo has no `CODE_OF_CONDUCT.md`, no `CODEOWNERS`, no `GOVERNANCE.md`, and no documented decision process. Outside contributors approaching the project today have no formal way to know who decides what, what behavior is expected, or how to escalate a stalled review.

For community delivery, the project needs a governance posture that is honest about its single-maintainer reality, sets behavioral expectations, and routes contributions reliably. It also needs to articulate a path toward multi-maintainer expansion without prematurely creating roles that have no occupants.

## Decision

Adopt a **Benevolent Dictator For Life (BDFL)** model with three concrete artifacts:

1. **`CODE_OF_CONDUCT.md`** — Contributor Covenant 2.1 verbatim, with the security email substituted for the enforcement contact. No bespoke language.
2. **`CODEOWNERS`** — A repo-root `CODEOWNERS` file routing every path to `@denisvieiradev`. Reviews are auto-requested by GitHub. Granular path ownership (e.g., `/packages/skills/` to a future skills maintainer) is left as a follow-up when there is a second maintainer to assign.
3. **`GOVERNANCE.md`** — Documents:
   - Current model: single BDFL with final decision authority.
   - Decision process: routine changes via PR review; significant changes (new package, breaking API, security-impacting design) via ADR in `.compozy/tasks/<name>/adrs/`.
   - SLA: best-effort PR triage within 7 days; bug acknowledgments within 14 days. No release SLA pre-1.0.
   - Path to co-maintainership: after 5 substantive merged PRs and 3 months of active participation, a contributor may be invited (BDFL's call) to become a committer; after 12 months, a committer may be invited to become a co-maintainer with merge rights for their domain.
   - Conflict resolution: the BDFL is the final tiebreaker. Disagreements should be raised in a GitHub Discussion and resolved publicly.
   - Succession: if the BDFL becomes inactive for 90 days without notice, named successors (initially empty; populated as co-maintainers are added) inherit decision authority.

## Alternatives Considered

### Alternative 1: Open multi-maintainer-ready from day one

- **Description**: Same as BDFL but `CODEOWNERS` uses `@gitwise-maintainers` team alias and `GOVERNANCE.md` reads as if there were multiple maintainers already.
- **Pros**: Pre-builds the structure for growth. Signals openness to contributors.
- **Cons**: Misrepresents the current reality. A team alias with one member adds bureaucracy without distributing load. Contributors who join expecting shared review burden discover otherwise.
- **Why rejected**: User selected BDFL. Honesty about bus-factor of 1 is more credible than aspirational governance.

### Alternative 2: README-note-only

- **Description**: Skip `GOVERNANCE.md` and `CODEOWNERS`; add a README section pointing at the security contact.
- **Pros**: Lowest overhead.
- **Cons**: No code-of-conduct (drops a baseline expectation for community-grade projects). No CODEOWNERS means review requests are not routed automatically — busy maintainer misses PRs. No documented escalation path.
- **Why rejected**: For full 1.0 hardening, this is below the floor.

### Alternative 3: Foundation-style governance with TSC

- **Description**: Adopt a Technical Steering Committee model with documented voting, quorum, and seat rotation.
- **Pros**: Most scalable. Familiar to enterprise contributors.
- **Cons**: Wildly disproportionate for a project of this size. Would create empty seats and unnecessary process.
- **Why rejected**: Premature.

## Consequences

### Positive

- Contributors arrive with clear behavioral expectations (Contributor Covenant) and a predictable review routing.
- The BDFL model matches the actual maintenance reality, so contributor expectations are calibrated.
- The path to co-maintainership is documented, removing the "how do I get commit access here" friction.
- `CODEOWNERS` reduces missed PRs by surfacing review requests in the maintainer's GitHub notifications.

### Negative

- The single maintainer is also the single source of CoC enforcement. Conflicts where the maintainer is involved have no neutral arbiter pre-co-maintainership.
- "Best-effort" SLAs are weaker than ideal, but accurate.

### Risks

- **Risk**: A CoC report involves the BDFL. **Mitigation**: documented escalation path to a named external reviewer (a peer maintainer from another related project; identified in `GOVERNANCE.md` as a placeholder and filled before community announcement).
- **Risk**: BDFL inactivity stalls the project. **Mitigation**: 90-day succession trigger and explicit grant of GitHub admin to a designated successor (initially "none — first co-maintainer to fill"); revisit at every release.
