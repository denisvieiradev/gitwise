# ADR-004: Transactional rollback for multi-step git workflows

## Status

Accepted

## Date

2026-05-21

## Context

The codebase audit identified three multi-step flows that can leave the user's repo in a partial state if interrupted:

1. **`gw release prepare` (gitflow)** — creates a release branch with `git checkout -b`. A later failure (package.json write error, API failure, etc.) leaves the branch orphaned. Retrying prepare hits `STRATEGY_RELEASE_BRANCH_EXISTS` and the user must manually delete the branch.
2. **Workspace version propagation** — `release.ts:1100–1150` writes the new version into each `packages/*/package.json`. If a write fails partway through, some packages are bumped and others are not. No rollback.
3. **Commit-split** — `commit.ts:295–302` resets staging then iterates `git add` + `git commit` for each planned commit. If any commit fails mid-loop, the repo has partial commits with no clean rollback path.

Today, every one of these failures is recoverable only by manual user intervention. As gitwise ships to a wider community — including users running it from CI, hooks, and unattended scripts — partial-state hazards become the most likely source of "gitwise corrupted my repo" issues. This ADR establishes the pattern for handling them.

## Decision

Introduce a `Transaction` primitive in `@denisvieiradev/gitwise-core` that records compensating actions as steps are applied, and runs them in reverse on failure.

```ts
type Step<T> = { name: string; apply: () => Promise<T>; compensate: (result: T) => Promise<void> };

export class Transaction {
  private applied: Array<{ step: Step<unknown>; result: unknown }> = [];

  async run<T>(step: Step<T>): Promise<T> {
    const result = await step.apply();
    this.applied.push({ step: step as Step<unknown>, result });
    return result;
  }

  async rollback(reason: GitwiseError): Promise<void> {
    for (const { step, result } of this.applied.reverse()) {
      try { await step.compensate(result); }
      catch (err) { logger.warn("compensate-failed", { step: step.name, reason: err }); }
    }
  }
}
```

Apply the pattern to the three identified flows:

1. **`release prepare` (gitflow)**: branch creation, gitignore mutation, plan write, and notes write are run as transaction steps with compensating `git branch -D`, gitignore revert, and `fs.unlink` actions. On failure, the transaction rolls back to the pre-prepare state. Plan file is only written after every other step succeeds.
2. **Workspace version propagation**: each `package.json` write captures the prior contents; compensate restores the prior bytes. Writes are still sequential (not concurrent) so ordering is deterministic.
3. **Commit-split**: each `git commit` step records the commit SHA; compensate runs `git reset --soft <prior-sha>` to remove only that commit while preserving the staged delta for retry. The pre-split working tree is captured as a stash compensating action at the start; if the stash compensate fires, it pops the stash to fully restore.

If `compensate` itself fails, the failure is logged with a `code: "ROLLBACK_PARTIAL"` warning, the user is shown a recovery hint pointing at `docs/recovery.md`, and the original error (with exit code from [[adr-003]]) is still surfaced. Rollback failures are non-fatal to the error path — they do not mask the original cause.

## Alternatives Considered

### Alternative 1: Idempotent retry (detect-and-resume)

- **Description**: Instead of rolling back, every flow becomes resumable. Prepare detects an existing release branch matching the planned target and continues from where it left off. Commit-split detects already-applied commits and skips them.
- **Pros**: Smaller surface area. No need for compensating actions. Aligns with how `git rebase --continue` works.
- **Cons**: Requires every step to be reliably observable from external state. Some steps (e.g., "API call to LLM happened") leave no externally visible footprint, so the resume logic must replay them — which costs tokens and time. Complexity moves from rollback into duplication detection, which is harder to test exhaustively.
- **Why rejected**: User selected the transactional option in clarification. Detect-and-resume is a fine fallback for cases where a compensate is impossible (e.g., a network push that succeeded but whose acknowledgement was lost) — the transactional pattern accommodates both: if `compensate` is a no-op, the flow degenerates to idempotent retry.

### Alternative 2: Document recovery + dry-run flag

- **Description**: Add `--dry-run` everywhere and a `TROUBLESHOOTING.md` listing manual recovery steps per failure mode. No new transactional code.
- **Pros**: Cheap.
- **Cons**: Pushes the burden onto the user. Does not scale as new flows are added.
- **Why rejected**: Inconsistent with the "Full 1.0 hardening" scope. A community-grade CLI should not require manual recovery for predictable failures.

### Alternative 3: Single-step atomic rewrites only

- **Description**: Restructure each multi-step flow into a single atomic operation (e.g., one large `git commit --amend` instead of per-commit splits, one giant package.json batch write via `git stash` + bulk-write + `git commit`).
- **Pros**: No need for a Transaction abstraction.
- **Cons**: Some flows fundamentally cannot be atomic (the API call must happen at a specific point; gitflow branch must exist before the commit). Loses observability — users want to see "step X failed" not "the whole operation failed."
- **Why rejected**: Not achievable for the identified flows without losing UX.

## Consequences

### Positive

- A failed `release prepare` leaves the repo exactly as it was before prepare started. Retries are unambiguous.
- Commit-split failures preserve user work: the prior tree is restored, the LLM-suggested plan is still in scope to retry.
- The Transaction primitive becomes available for future multi-step flows (e.g., a planned "interactive rebase assistant") without re-deriving the pattern.
- Test coverage targets become concrete: every flow's failure modes are enumerable as "step N fails" scenarios.

### Negative

- Each flow takes longer to implement and test. Roughly 1.5x the lines compared to a non-transactional version.
- Compensating actions for git operations occasionally fail (e.g., a branch that another process is checked out on cannot be deleted). The `ROLLBACK_PARTIAL` path mitigates but does not eliminate this.
- The abstraction adds cognitive load for new contributors. `CONTRIBUTING.md` will need a short "how to write a transactional flow" section.

### Risks

- **Risk**: A compensate action introduces its own bug (e.g., wrong SHA in reset), making rollback worse than no rollback. **Mitigation**: every transactional step is unit-tested for its apply path AND its compensate path. Integration tests deliberately fail at each step boundary and assert end-state equality with the pre-apply state.
- **Risk**: Concurrent gitwise invocations on the same repo race on the same release plan file or working tree. **Mitigation**: a lockfile at `.gitwise/.lock` is acquired at the start of any flow with side effects; conflicting invocations fail fast with `code: "REPO_LOCKED"`.
- **Risk**: Performance regression on the happy path due to compensating-action setup overhead. **Mitigation**: compensating actions are lazy — only captured state needed for compensate is materialized; the apply path itself is unchanged.

## Implementation Notes

- Add `packages/core/src/infra/transaction.ts` with the `Transaction` class.
- Add `packages/core/src/infra/lockfile.ts` for the `.gitwise/.lock` advisory lock (PID + timestamp; stale-lock detection after 10 minutes).
- Migrate flows in this order: (1) workspace version propagation (simplest), (2) gitflow prepare, (3) commit-split (most complex).
- Add a `docs/recovery.md` listing what to do when `ROLLBACK_PARTIAL` is reported.
- Reuse `GitwiseError.code` ([[adr-003]]) to drive rollback decisions where a partial rollback is appropriate (e.g., `code: "PUSH_FAILED"` after a successful tag should roll back the tag but not the bump commit).
