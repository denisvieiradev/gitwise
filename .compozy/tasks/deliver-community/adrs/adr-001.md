# ADR-001: Supply-chain integrity via npm provenance, OIDC, signed tags, and SBOM

## Status

Accepted

## Date

2026-05-21

## Context

`gitwise` ships three publishable packages to npm (`@denisvieiradev/gitwise-core`, `@denisvieiradev/gitwise`, `@denisvieiradev/gitwise-skills`). The current release pipeline (`.github/workflows/release.yml`) uses a long-lived `NPM_TOKEN` secret and publishes without provenance. Tags are unsigned. There is no SBOM. For an open-source project that will be invoked from inside other repos and execute git commands on user machines, downstream consumers cannot verify what was built from what source, nor pin to a known-good supply chain.

Pre-1.0 hardening (see scope decision in techspec) requires a credible supply-chain story before inviting outside contributors and broader adoption.

## Decision

Adopt the following supply-chain controls for every release:

1. **npm provenance via OIDC** — Replace `NPM_TOKEN` with GitHub Actions OIDC trust. Publish with `npm publish --provenance --access public` from `release.yml`. Provenance attestations link each published tarball to the exact GitHub Actions run and commit SHA that produced it.
2. **Signed git tags** — Release tags are signed with a maintainer GPG key (`git tag -s`). The public key is published in `SECURITY.md` and in a `KEYS.asc` file at the repo root.
3. **CycloneDX SBOM per release** — `release.yml` runs `@cyclonedx/cdxgen` after the build, producing `sbom-<version>.cdx.json`. The SBOM is uploaded as a GitHub Release asset and attested via the GitHub artifact-attestations action.
4. **Pinned GitHub Actions** — All third-party Actions in workflows are pinned to commit SHAs with the version tag in a comment (e.g., `uses: actions/checkout@<sha>  # v4`). Dependabot is configured to PR new pins.

## Alternatives Considered

### Alternative 1: SLSA Build Level 3 with slsa-github-generator

- **Description**: Use the official `slsa-github-generator` reusable workflow to produce SLSA L3 provenance with hermetic builds in an ephemeral runner.
- **Pros**: Highest assurance level. Strongest defense against compromised build environments. Matches what large-scale tooling teams adopt.
- **Cons**: Significant CI complexity. The reusable workflow has its own learning curve and quirks. Build time roughly doubles. Many open-source tools of similar size have not adopted L3.
- **Why rejected**: Disproportionate cost for the current threat model. SLSA L3 is a follow-up after 1.0 ships if the project's reach grows.

### Alternative 2: Provenance-only (no Sigstore, no SBOM)

- **Description**: Just add `--provenance` to `npm publish` and switch to OIDC. Leave tags unsigned, no SBOM, no attestations.
- **Pros**: Smallest delta from the current pipeline. Lowest maintenance overhead.
- **Cons**: Downstream consumers cannot verify the dependency graph of a release. No cryptographic linkage from git tag to published artifact for consumers who do not trust npm. Misses half the SLSA L2 controls.
- **Why rejected**: User selected the broader option in techspec clarification. Provenance alone is the floor, not the bar gitwise should set as it invites external contributions.

### Alternative 3: Keep NPM_TOKEN, add provenance

- **Description**: Provenance with the existing `NPM_TOKEN`-based auth.
- **Pros**: Minor change.
- **Cons**: `NPM_TOKEN` remains a long-lived secret with publish scope — a top supply-chain compromise vector. OIDC eliminates the secret entirely.
- **Why rejected**: OIDC removes a high-value attack surface and is now the npm-recommended pattern.

## Consequences

### Positive

- Consumers can verify, with one `npm view <pkg> --json | jq .dist.attestations`, that any released version was built from a specific public commit by a specific public workflow.
- Compromised CI credentials no longer enable publish: OIDC tokens are scoped per workflow run.
- SBOM gives downstream consumers a machine-readable dependency manifest for vulnerability triage.
- Signed tags let cautious users verify history without trusting GitHub.

### Negative

- Release workflow becomes longer and more steps can fail. Most failure modes (OIDC misconfig, cdxgen install failure) are recoverable by re-running the workflow.
- Contributors who fork and publish their own builds get unsigned, no-provenance artifacts — `gitwise` does not officially sanction forked releases.
- Maintainer must hold and rotate a GPG key. Key compromise procedure documented in `SECURITY.md`.

### Risks

- **Risk**: OIDC trust misconfiguration could allow another workflow in the org to publish. **Mitigation**: trust policy on the npm side scopes to `denisvieiradev/gitwise` + branch `main` + workflow `release.yml`.
- **Risk**: SBOM generation transient failures break releases. **Mitigation**: SBOM is generated and uploaded but is not gated as a publish blocker for now; failure surfaces a non-fatal warning.
- **Risk**: Maintainer GPG key loss disrupts releases. **Mitigation**: store the key in two secured locations (password manager + offline backup). Publish a key-rotation procedure.

## Implementation Notes

- Update `.github/workflows/release.yml`:
  - Add `permissions: { id-token: write, contents: write }` to the publish job.
  - Replace `npm publish --workspaces` with a step that publishes each workspace explicitly with `--provenance`.
  - Add a step that runs `npx @cyclonedx/cdxgen -t npm -o sbom-${VERSION}.cdx.json .`.
  - Add a step that uploads the SBOM via `gh release upload`.
- Update `release.ts` to require `signTags: true` by default; expose a `--no-sign` escape hatch documented as testing-only.
- Add `KEYS.asc` to the repo with the maintainer's public key fingerprint linked from `SECURITY.md`.
