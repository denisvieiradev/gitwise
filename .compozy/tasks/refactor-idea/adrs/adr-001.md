# ADR-001: gitwise will ship as an orthogonal four-command AI git toolbelt

## Status

Accepted

## Date

2026-05-16

## Context

The current product (`devflow-cli`) is an end-to-end PRD → techspec → tasks → run-tasks → review → PR → release pipeline. The user has decided to pivot to a focused product called `gitwise`, scoped only as "a Git assistant boosted with AI." The new product must be installable in two equally-supported modes: as a Claude Code skills bundle, and as a standalone CLI (`gw`) that connects to Claude (preferring an installed Claude Code, falling back to a `ANTHROPIC_API_KEY`).

Three product shapes were considered for this new product:

- A four-command toolbelt with no required workflow (each command independent).
- A four-command toolbelt plus a `gw ship` orchestrator that chains commit → review → pr (→ release).
- A skills-first design where the CLI mirrors the skills 1:1, with shared logic underneath.

The decision had to account for: parity between install modes (chosen separately), the existing strength of the multi-context commit logic in the current codebase, and the explicit goal of dropping the heavyweight pipeline.

## Decision

Ship `gitwise` as four orthogonal commands — `gw commit`, `gw review`, `gw pr`, `gw release` — with no required workflow state and no `init` command. Each command operates standalone and is published as both a CLI subcommand and a discrete Claude Code skill, providing day-one feature parity across both install modes.

Multi-context commit splitting (auto-grouping a noisy staged diff into N logical conventional commits) is the headline differentiator and lives inside `gw commit`. The broader pre-ship breadth (`commit` → `review` → `pr` → `release`) is the secondary positioning, but it is achieved by users composing commands themselves, not by a built-in orchestrator.

## Alternatives Considered

### Alternative 1: Four commands + `gw ship` orchestrator

- **Description**: Same four commands, plus a `gw ship` command that chains commit → review → pr (and optionally release) interactively, persisting light cross-command state in `.gitwise/`.
- **Pros**: Stronger headline narrative ("from staged diff to merged PR in one command"); chained context can produce richer PR descriptions that reference the commit reasoning; clearer differentiation from single-purpose competitors.
- **Cons**: Reintroduces local state, which is the exact concept being pruned from devflow-cli; couples commands and risks reproducing the "too-prescribed workflow" problem the pivot is trying to solve; harder to map onto independent skills.
- **Why rejected**: The user explicitly chose true parity between skills and CLI install modes (which strongly favors 1-command-to-1-skill mapping), and explicitly chose to keep all four current commands without adding new orchestration. A `gw ship` style chain can be added later in phase 2 if usage data warrants it.

### Alternative 2: Skills-native, CLI as mirror

- **Description**: Treat Claude Code skills as the primary design surface (markdown spec + tool allowlist) and ship the CLI as a thin mirror, with the shared multi-context-commit logic sitting beneath both.
- **Pros**: Forces a clean separation between "AI prompt/skill logic" and "execution glue"; skills can leverage Claude Code's richer context (open files, conversation history) for better suggestions; CLI stays small.
- **Cons**: Higher upfront architecture cost; skill authoring is a newer, less-stable discipline than CLI authoring; real risk of feature divergence between the two surfaces over time.
- **Why rejected**: Day-one parity is the goal, not skill-primacy. The codebase already has strong CLI scaffolding and weak skills scaffolding; rebuilding around skills-first would slow MVP without proportional user benefit. The chosen approach achieves parity by treating the underlying TypeScript modules as the source of truth, with both surfaces calling into them.

## Consequences

### Positive

- One-to-one mapping between commands and skills simplifies parity, testing, documentation, and the install experience for both modes.
- No required initialization (`gw init` is not in the MVP) lowers adoption friction — users run `npm i -g @denisvieiradev/gitwise` and immediately try `gw commit`.
- Reuses the strongest part of the existing devflow-cli codebase (multi-context commit logic, git/GitHub infra, model routing) while shedding the pipeline state that motivated the rename.
- Each command can ship, version, and be deprecated independently.

### Negative

- No automatic cross-command context — a richer PR body that incorporates commit reasoning requires the user to invoke commands in sequence (or for a future `gw ship` to be added).
- Some users coming from the "pipeline" mental model of devflow-cli will need a moment to adjust to the toolbelt framing.

### Risks

- Risk: feature drift between skills and CLI modes. Mitigation: shared TypeScript modules are the canonical source; skills are thin wrappers that call into them.
- Risk: positioning collision with the official Claude Code "commit-commands" plugin and other commit-only competitors. Mitigation: lead marketing with multi-context commit splitting (a clear gap), and the four-command breadth (review, pr, release) as a moat.
- Risk: dropping `init` removes auto-detection of project language/framework/test runner. Mitigation: detect on-demand inside each command from `package.json` / file heuristics, fall back to sensible defaults, and allow override via optional repo-level `.gitwise.json`.

## Implementation Notes

- New npm package: `@denisvieiradev/gitwise`, binary `gw`. New git repository (separate from `devflow-cli`).
- Carry over: `src/commands/commit.ts`, `review.ts`, `pr.ts`, `release.ts`; `src/infra/git.ts`, `github.ts`; `src/providers/claude.ts`, `claude-code.ts`, `factory.ts`, `model-router.ts`; `src/core/config.ts`, `template.ts`; templates `commit.md`, `pr.md`, `release-*.md`.
- Remove: `init`, `prd`, `techspec`, `tasks`, `run-tasks`, `test`, `done`, `status` commands; `src/core/pipeline.ts`, `state.ts` (feature-tracking), `context.ts`, `drift.ts`; templates `prd.md`, `techspec.md`, `tasks.md`.
- Auth resolver order in CLI: detect installed Claude Code → use as subprocess provider; else require `ANTHROPIC_API_KEY` (env var or one-time `gw auth login` flow, deferred to phase 2 if env-only suffices for MVP).
- Skills bundle: ship four skill markdown files (one per command) under a `gitwise` plugin namespace; each skill's tool allowlist matches what its CLI counterpart needs (read files, run git, optionally `gh`).
- `devflow-cli` is archived with a final release that points users to gitwise; no in-place rename, no dual-binary alias.
