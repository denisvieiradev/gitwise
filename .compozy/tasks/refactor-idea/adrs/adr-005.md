# ADR-005: Locked-version monorepo releases via dogfooded `gw release`

## Status

Accepted

## Date

2026-05-16

## Context

The monorepo ships three packages (`core`, `cli`, `skills`). They must stay coherent — a `core` change that the `cli` depends on must land in users' machines together, not in separate releases. We also want the release tool itself to be `gw release`: we are building an AI git assistant whose flagship features include automated release notes and versioning, so dogfooding signals confidence and lowers the maintainer's per-release effort.

Dogfooding introduces a bootstrap problem: `gw release` does not exist on day one of the project. We need a path from "manual releases while building" to "`gw release` runs the release."

## Decision

Use a **locked, shared version across all packages**. Every release bumps `core`, `cli`, and `skills` to the same version. The release process has two phases:

**Phase 0 — Bootstrap (versions `0.0.x` through `0.x.y`).**
Releases are cut manually:

- Bump root `package.json` version.
- Use a small Node script (`scripts/release.mjs`) to propagate the version to each `packages/*/package.json`.
- `npm publish --workspaces` from CI on tag push.
- Hand-write `CHANGELOG.md` and `releases/<version>.md`.

Phase 0 ends when `gw release` is implemented, tested against the gitwise monorepo itself, and passes a real release end-to-end in a staging branch.

**Phase 1 — Dogfooded (from version `0.1.0` or `1.0.0` onward).**
Releases are cut by running `gw release` from the monorepo root:

- `gw release` reads `package.json`, recommends a semver bump from commit history, generates the changelog and release notes.
- It is configured (via a `.gitwise.json` at the repo root) to propagate the new version to all workspace packages before committing.
- Tagging, pushing, and `gh release create` are unchanged.

CI publishes on tag push: every tag that matches `v*` runs `npm publish --workspaces --access public`.

## Alternatives Considered

### Alternative 1: Changesets-based independent versioning

- **Description**: Adopt `@changesets/cli`. Contributors add changesets; CI bumps each package independently and publishes on merge.
- **Pros**: Industry-standard for npm monorepos; granular versioning; supports prereleases out of the box.
- **Cons**: Adds a tool we'd be paid in dependency surface; independent versioning is an over-correction when packages always release together; doesn't dogfood `gw release`.
- **Why rejected**: Coupling is the desired property here, not a bug. Three packages always change together by design.

### Alternative 2: Manual versioning forever

- **Description**: Skip the `gw release` dogfooding plan; permanently bump each `package.json` by hand.
- **Pros**: Lowest tooling investment.
- **Cons**: Misses the most visible product validation we have — using our own release tool to release the product. Error-prone over time.
- **Why rejected**: We are building a release tool. Not using it on ourselves is a credibility tax.

### Alternative 3: Independent versions with shared release flow

- **Description**: Each package versions independently but releases together (compute per-package bump from commit history).
- **Pros**: More flexibility if `skills` and `cli` diverge in API.
- **Cons**: Higher implementation burden for `gw release`; harder to communicate "what version am I on" to users.
- **Why rejected**: Defer until evidence shows the locked-version model is causing problems.

## Consequences

### Positive

- Users (and `gw config` outputs) display one version number across all surfaces.
- Bootstrap path is concrete and time-bounded — no eternal "we'll dogfood soon."
- The release tool becomes its own first power user, exposing UX gaps fast.

### Negative

- A skills-only bugfix forces a `cli` + `core` version bump that contains no changes. Minor `package-lock.json` churn; otherwise harmless.
- During Phase 0, two release paths exist (manual script + the in-progress `gw release` command). Brief duplication.

### Risks

- Risk: `gw release` breaks the monorepo's own release flow once dogfooded. Mitigation: keep `scripts/release.mjs` in the repo as a documented fallback; add a CI smoke test that runs `gw release --dry-run` on every merge to main.
- Risk: contributors confused by which path to use during Phase 0. Mitigation: a `CONTRIBUTING.md` section names Phase 0 explicitly and points to the active script.

## Implementation Notes

`scripts/release.mjs` (Phase 0) responsibilities:

- Accept a bump argument (`patch`/`minor`/`major`) or an explicit version.
- Update root `package.json`.
- Walk `packages/*/package.json` and set the same version, plus update `workspace:*` references if any get pinned.
- Stage, commit (`chore(release): vX.Y.Z`), tag (`vX.Y.Z`), push.
- CI on tag push runs `npm publish --workspaces --access public`.

Phase 1 (`gw release` from repo root) reuses the same publishing CI; the only change is that humans no longer run `scripts/release.mjs` — `gw release` does the version propagation, changelog, and notes itself.
