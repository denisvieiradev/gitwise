# ADR-003: Non-interactive core with four high-level command functions

## Status

Accepted

## Date

2026-05-16

## Context

Day-one install parity (CLI + Claude Code skills) requires a single source of truth for what each command does. The CLI presents interactive prompts via `@clack/prompts` to confirm split plans, edit messages, and pick files. The skills bundle runs inside Claude Code, where the interactive dialog is driven by Claude itself, not by an OS-level prompt library. The core must serve both surfaces without forcing one to fake the other's interaction model.

A second design pressure: each command should benefit from optional user-supplied intent (e.g., `gw commit "this fixes the OAuth callback bug"`) so the LLM has more context than the raw diff. This must work uniformly in CLI and skills.

## Decision

`@denisvieiradev/gitwise-core` exposes **four async, non-interactive functions** — `commit()`, `review()`, `pr()`, `release()` — that take a typed options object and return a typed result object (plan/draft). The functions never block on user input; if a decision is needed (e.g., split vs. single commit), they include the options in the returned plan and let the caller orchestrate confirmation.

The CLI wraps each call with `@clack/prompts` to present plans, accept edits, and confirm execution. Defaults are tuned so a flag-free invocation produces useful output for daily use; every default can be overridden via flags (e.g., `--no-split`, `--no-confirm`, `--message`, `--push`).

The skills bundle calls the same core functions from small Node scripts and returns the plan as structured markdown or JSON to Claude Code, which uses its own conversation surface to confirm choices with the user.

Every command function accepts an optional `prompt` string field on its options object for free-form user intent. The CLI exposes this as a positional argument (`gw commit "intent text"`); the skill passes it from the user's Claude Code message.

## Alternatives Considered

### Alternative 1: Inject an `IO` adapter

- **Description**: Core functions take an `io: IO` parameter with `confirm()`, `select()`, `text()` methods. CLI passes a clack-backed IO; skills pass a Claude-aware IO.
- **Pros**: Maximum reuse — the core orchestrates the full interactive flow once.
- **Cons**: Forces the skills-mode IO to either reimplement Claude Code's prompting (re-routing through MCP-style asks) or no-op everything, which collapses to the chosen design anyway. Adds an abstraction layer.
- **Why rejected**: User explicitly preferred non-interactive core: "Always use defaults to make daily use easier, but allow the user to pass a command to override them if necessary during execution." Cleaner than building a parallel interaction abstraction that one side largely ignores.

### Alternative 2: Two entry points per command

- **Description**: Core exports `commit.interactive()` (clack-driven) and `commit.dryRun()` (data-returning) for each command.
- **Pros**: Caller picks the right one explicitly.
- **Cons**: Two near-duplicate code paths per command; duplicated argument plumbing; doubled maintenance.
- **Why rejected**: YAGNI. One non-interactive function plus a thin CLI wrapper is smaller and clearer.

### Alternative 3: Low-level primitives only

- **Description**: Core exposes `getDiff()`, `generateCommitMessage()`, `splitDiffIntoCommits()`, etc. CLI and skills orchestrate.
- **Pros**: Maximum flexibility.
- **Cons**: Orchestration logic duplicated between CLI and skills — the exact divergence risk the monorepo is designed to prevent.
- **Why rejected**: Pushes the most error-prone code (multi-context detection, commit-plan assembly) to the consumers.

## Consequences

### Positive

- Single non-interactive function per command means parity is structural, not policed.
- The CLI's only responsibility is argument parsing + prompt rendering, keeping `packages/cli` small.
- Skills scripts are trivial: parse arguments → call core → emit markdown.
- Free-form `prompt` field gives users a single, consistent way to convey intent across both modes and across all four commands.

### Negative

- The CLI must implement its own clack-based confirmation rendering for plans/drafts. Initial implementation cost is borne in `packages/cli` rather than amortized in core.
- Skills mode delegates dialog to Claude. If Claude makes a wrong default choice, the user must explicitly correct it. Mitigation: command flags (`--no-split`) are documented as overrides in the skill instructions.

### Risks

- Risk: skills-mode output and CLI-mode output drift over time as one surface gains flags the other doesn't expose.
  - Mitigation: every option on the core function is automatically reachable from both surfaces — CLI exposes it as a flag; skill instructions document it as a knob Claude can pass.
- Risk: returning rich result objects to the CLI invites coupling on internal shapes.
  - Mitigation: result types are explicit and exported from `@gitwise/core`; treated as semver-relevant public API.

## Implementation Notes

Sketch of the core API:

```typescript
// packages/core/src/index.ts
export interface CommitOptions {
  cwd?: string;
  prompt?: string;          // free-form user intent
  split?: "auto" | "never" | "always";
  push?: boolean;
  baseBranch?: string;
  io?: ProviderConfig;      // override provider for this call
}

export interface CommitPlan {
  type: "single" | "split";
  commits: Array<{
    message: string;
    description?: string;
    files: string[];
  }>;
  tokens: { input: number; output: number };
}

export async function commit(opts: CommitOptions): Promise<CommitPlan>;
```

CLI flow (sketch):

```typescript
// packages/cli/src/commands/commit.ts
const plan = await commit({ prompt: posArg, push, split, baseBranch });
const choice = await p.select({ ... }); // present plan to user
if (choice === "execute") await applyCommitPlan(plan);
```

Skill flow (sketch):

```typescript
// packages/skills/scripts/commit.ts
const plan = await commit({ prompt: process.argv[2] });
console.log(formatPlanAsMarkdown(plan));
// Claude Code reads stdout and continues the conversation
```
