# ADR-004: Explicit first-run provider choice with persisted user config

## Status

Accepted

## Date

2026-05-16

## Context

The PRD requires `gw` to "connect with Claude" via two paths: Claude Code subprocess (zero-key auth, inherits Claude Code's session) or direct Anthropic API access (`ANTHROPIC_API_KEY`). Both providers exist in the predecessor codebase (`devflow-cli`) and have proven retry/error semantics.

Three resolution strategies were on the table: silent auto-detection on every run, auto-detect with cache, and explicit-once-then-persist. The product wants zero-init for the toolbelt feel, but also wants the choice to be transparent rather than something that silently changes when the user later installs/uninstalls Claude Code.

## Decision

On the first invocation of any `gw <command>` (detected by absence of `~/.gitwise/config.json`), prompt the user once:

1. If `claude` is detected on PATH, offer it as the default option.
2. Otherwise, prompt for `ANTHROPIC_API_KEY` (or instruct the user to set it as an env var).
3. Persist the choice (`provider: "claude-code" | "api"`) and any API-key reference into `~/.gitwise/config.json`.

After first run, the resolver reads the persisted provider and never re-prompts. Users change the provider via `gw config provider <value>` (or by editing the file directly). Both providers (`packages/core/src/providers/claude-code.ts` and `packages/core/src/providers/anthropic.ts`) are bundled in core; the factory selects based on config.

API keys are read from `ANTHROPIC_API_KEY` env var first; if absent and `provider === "api"`, the resolver checks `~/.gitwise/.env` (created with `0600` perms during first run). Keys are never written to `config.json` itself.

## Alternatives Considered

### Alternative 1: Auto-detect on every invocation

- **Description**: Each `gw` run probes for `claude` on PATH and chooses on the fly.
- **Pros**: Truly zero-config; tracks user environment changes transparently.
- **Cons**: Silent behavior changes when the user installs/uninstalls Claude Code; the same staged diff might use different providers across runs; harder to debug "why did this output look different."
- **Why rejected**: Hidden state changes are exactly the kind of surprise the toolbelt framing should avoid.

### Alternative 2: Auto-detect with session cache

- **Description**: Detect once per day/session; cache resolved provider in `~/.gitwise/state.json`.
- **Pros**: Marginal startup-time savings on repeat calls.
- **Cons**: Adds a cache invalidation problem; doesn't solve the "hidden state" critique. PATH lookups are sub-millisecond anyway.
- **Why rejected**: Complexity without proportional benefit.

### Alternative 3: API-only in CLI, Claude Code-only in skills

- **Description**: Hard separation — the CLI always requires `ANTHROPIC_API_KEY`; the skills bundle always uses Claude Code's auth.
- **Pros**: Smallest code; clearest mental model per surface.
- **Cons**: Forfeits "CLI is free if Claude Code is installed" — a real adoption-wedge advantage; punishes the most-likely user (someone who has Claude Code and wants the CLI too).
- **Why rejected**: Deletes a major value prop of the dual-install design.

## Consequences

### Positive

- One explicit choice at install time → predictable behavior forever after.
- `gw config provider <value>` gives users a clear "switch" without forcing reinstall.
- Both providers stay alive in core, which lets us route command tiers (per `model-router.ts`) the same way they did in devflow.
- API-key handling stays separate from `config.json`, keeping that file safe to commit if a user ever wants a public sample.

### Negative

- One-time friction on the first command run. Mitigation: the prompt is short (Claude Code detected? Y/n) and accepts an `--api-key` flag on the first command to skip the interactive flow entirely.
- Stale config if the user uninstalls Claude Code while still configured for it. Mitigation: when `provider === "claude-code"` but `claude` is not found, fail with a single line: "claude not found; run `gw config provider api` to switch."

### Risks

- Risk: API key leakage if users accidentally commit `~/.gitwise/.env`. Mitigation: file is under the user's home directory, not in the repo; permissions set to `0600`; documentation notes never copy this file into a repo.
- Risk: Claude Code binary path resolution fails on uncommon installs (e.g., Snap on Linux). Mitigation: port the multi-strategy resolver from devflow's `claude-code.ts` (PATH → Homebrew → nvm); document a `claudeCliPath` config override.

## Implementation Notes

Schema for `~/.gitwise/config.json`:

```typescript
interface UserConfig {
  provider: "claude-code" | "api";
  claudeCliPath?: string;           // override binary resolution
  models: {
    fast: string;                   // default: "claude-haiku-4-5-20251001"
    balanced: string;               // default: "claude-sonnet-4-6"
    powerful: string;               // default: "claude-opus-4-7"
  };
  language: "en" | "pt" | "es" | "fr";
  defaultBaseBranch?: string;       // default: auto-detect
  commitConvention: "conventional" | "free";
}
```

First-run flow (in `packages/cli/src/first-run.ts`):

1. If `fs.existsSync("~/.gitwise/config.json")` → return early.
2. Try to resolve `claude` binary (PATH, Homebrew, nvm). Cache the result.
3. Print a single welcome banner; then `p.select({ ... })` with the detected option as default.
4. If `api`: `p.password({ message: "Anthropic API key" })`. Write to `~/.gitwise/.env` (mode `0600`).
5. Write `~/.gitwise/config.json` with the chosen provider + default models + language.
6. Continue with the original command invocation.

Reconfiguration via `gw config <key> <value>`:

```typescript
gw config provider api
gw config models.balanced claude-sonnet-4-6
gw config language pt
```
