# ADR-002: Monorepo with npm workspaces (core / cli / skills)

## Status

Accepted

## Date

2026-05-16

## Context

The product (gitwise) ships in two install modes with day-one parity: the `gw` CLI (global npm install) and a Claude Code skills bundle (per-project plugin install). Both surfaces must produce identical outputs from identical inputs. The longer-term plan is to add additional AI-assistant hosts (Copilot, Codex, Cursor) as separate adapter packages without rewriting the core.

The competing concerns:
- Avoid logic duplication between CLI and skills.
- Keep each install artifact small (skills users should not pull CLI-only deps; CLI users should not pull skill manifests).
- Make future host adapters cheap to add.
- Use tooling familiar to TypeScript/Node developers.

## Decision

Adopt a single git repo (`gitwise`) using **npm workspaces** with three publishable packages:

- `@denisvieiradev/gitwise-core` — shared logic: prompt templates, git/github primitives, LLM providers, the four high-level command functions (`commit`, `review`, `pr`, `release`). Exposes a `MockLLMProvider` from a `testing` subpath for downstream tests. No CLI, no skill manifests.
- `@denisvieiradev/gitwise` — the `gw` CLI binary. Thin commander + @clack/prompts layer that calls into `gitwise-core`.
- `@denisvieiradev/gitwise-skills` — the Claude Code plugin (manifest + skill markdown files + tiny Node scripts in `packages/skills/scripts/*.ts` that import `gitwise-core`). Installable per-project as a Claude Code plugin without requiring the CLI to be on PATH.

All three packages share a locked version (see [ADR-005](adr-005.md)).

## Alternatives Considered

### Alternative 1: Single package, two surfaces

- **Description**: One npm package ships both the CLI binary and `skills/` manifests + scripts.
- **Pros**: One artifact; simplest publish flow; no internal dependency wiring.
- **Cons**: Skills users pull CLI dependencies (`commander`, `@clack/prompts`) they don't need; harder to add future host adapters without bloating the package; mixes concerns.
- **Why rejected**: Forces every install mode to carry every dep; misaligns with the parity + extensibility goal.

### Alternative 2: Two separate git repos

- **Description**: `gitwise` (CLI) and `gitwise-skills` (Claude Code plugin) in different repos.
- **Pros**: Maximum decoupling.
- **Cons**: Doubles release cadence; cross-cutting changes need two PRs; high drift risk between modes; no shared `core`.
- **Why rejected**: Drift between modes directly threatens the parity claim.

### Alternative 3: One adapter package per host upfront (`packages/hosts/*`)

- **Description**: Scaffold `packages/hosts/claude-code`, `packages/hosts/copilot`, `packages/hosts/cursor`, `packages/hosts/codex` at MVP.
- **Pros**: Future-proof structure.
- **Cons**: YAGNI violation; only Claude Code is being implemented now.
- **Why rejected**: User explicitly chose "Claude-only for MVP, refactor later" (see [ADR-003](adr-003.md)). The monorepo layout below trivially supports adding `packages/hosts/*` later without disrupting `core` or `cli`.

## Consequences

### Positive

- Single source of truth for prompts, providers, git logic. Both surfaces stay in sync by construction.
- Minimal per-package dependencies. Skills users never pull `commander`; CLI users never pull skill manifest tooling.
- Future host adapters (`packages/hosts/copilot`, etc.) drop into the workspace without restructuring.
- Existing test infrastructure (Jest + ts-jest ESM) ports cleanly to per-package `__tests__/` folders.

### Negative

- Three packages to keep in sync. Mitigated by locked versions ([ADR-005](adr-005.md)).
- Slightly higher friction for first-time contributors who haven't seen npm workspaces before.

### Risks

- Risk: circular references between packages. Mitigation: only `cli` and `skills` depend on `core`; never the reverse.
- Risk: published package contents accidentally include monorepo-only files (e.g., test fixtures, root `.gitignore`). Mitigation: each package has an explicit `files` field in its `package.json` and a publish smoke test in CI.

## Implementation Notes

Workspace layout:

```
gitwise/                        (git root, private root package.json)
├── package.json                workspaces: ["packages/*"]
├── tsconfig.base.json
├── tsup.config.ts              shared bundle config (per-package overrides)
├── jest.config.ts              shared Jest config
├── packages/
│   ├── core/
│   │   ├── package.json        @denisvieiradev/gitwise-core
│   │   ├── src/                index.ts, commands/, providers/, infra/, templates/
│   │   ├── templates/          commit.md, pr.md, review.md, release-*.md
│   │   └── __tests__/
│   ├── cli/
│   │   ├── package.json        @denisvieiradev/gitwise (bin: gw → dist/index.js)
│   │   ├── src/                index.ts, program.ts, commands/
│   │   └── __tests__/
│   └── skills/
│       ├── package.json        @denisvieiradev/gitwise-skills
│       ├── plugin.json         Claude Code plugin manifest
│       ├── skills/             commit.md, review.md, pr.md, release.md (skill specs)
│       ├── scripts/            commit.ts, review.ts, pr.ts, release.ts (thin Node)
│       └── __tests__/
└── .compozy/                   (existing — PRD/tasks for this refactor)
```

Inter-package linking via workspace protocol: `"@denisvieiradev/gitwise-core": "workspace:*"` in `cli` and `skills`.
